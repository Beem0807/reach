# Reach API Reference

All API calls use HTTPS. The base URL is whatever you set as `API_URL` during deployment.

---

## Authentication

Four types of credentials:

| Credential                | How to get it                                            | Used for                                                                          |
| ------------------------- | -------------------------------------------------------- | --------------------------------------------------------------------------------- |
| Admin session token       | `POST /admin/login` with `ADMIN_PASSWORD`                | Platform admin **console** (`/admin/*`) - tenant provisioning, cross-tenant users |
| Tenant session token      | `POST /tenant/login` with username + password            | Tenant **console** (`/tenant/*`) - the web UI                                     |
| API token (`tok_...`)     | Tenant console → **API Tokens → New token**              | **CLI and MCP server** - the operational tenant surface (see scope below)         |
| Agent token (`agent_...`) | Issued at claim time, managed automatically by the agent | Agent sync, job results                                                           |

Both session tokens expire after 8 hours; the console reissues them on login. API tokens do not expire automatically - revoke them explicitly when no longer needed.

**API-token scope.** An API token authenticates as its owning user (with that user's **role**), and works on the CLI endpoints (`/me`, `/jobs`, `/agents`, `/approvals/*`, `/fleets/*`) **and** the operational tenant endpoints - `/tenant/agents/*`, `/tenant/fleets/*`, and `/tenant/approvals/*` (create/approve/deny, role-permitting). It is **rejected** on the sensitive console tier - user management (`/tenant/users/*`), API-token management (`/tenant/api-tokens/*`), tenant settings (`/tenant/settings`), and audit logs (`/tenant/audit-logs`, `/admin/*`) - which require a real session login. So an operator's API token can approve a request from the CLI, but no API token can create users or mint/revoke tokens. The user's role is always enforced regardless of token type.

> **Rate limits** are enforced on every endpoint in the **Docker / FastAPI** deployment and shown in the `Rate limit` column of each table below. Lambda relies on API Gateway's built-in throttling instead. The rate key is the Bearer token when present (API token, session token, or agent token), falling back to client IP. See [Rate limits](#rate-limits) for the full model.

## Platform admin endpoints

Require an admin session token obtained via `POST /admin/login`.

**Scope**: tenant provisioning and cross-tenant user management. The platform admin cannot manage agents, policies, or approvals - those belong to a tenant's operators and admins.

### Authentication

```
POST /admin/login
Body: {"password": "<ADMIN_PASSWORD>"}
Response: {"token": "<session_token>", "expires_at": "..."}
```

Rate limit: **10/min per IP**.

These endpoints back the **web console**. They are authenticated with the **console session
token** returned by `POST /tenant/login` (an 8-hour token, distinct from an API token). The
`Auth` column shows the **minimum role** required.

The **CLI and MCP server do not use these** - they authenticate with an **API token** (`tok_…`,
created in the console under **API Tokens**) and call the smaller [User (CLI) endpoints](#user-cli-endpoints)
below (`/me`, `/jobs`, `/agents`, `/approvals/pending`).

Role determines access level within the tenant:

| Role        | Can do                                                                                              |
| ----------- | --------------------------------------------------------------------------------------------------- |
| `developer` | Submit jobs, view agents, request/view approvals, manage own API tokens                             |
| `operator`  | All developer ops + manage agents (create, revoke, policy mode, tags) + review and manage approvals |
| `admin`     | All operator ops + manage users and view audit logs                                                 |

**Agent-level access.** On top of role, non-admin users are scoped to a subset of agents/fleets,
**read-only or read-write** (see `PUT /tenant/users/{id}/agents`). Non-admins start with **no
access by default**. Read access is the union of the `readwrite_*` and `readonly_*` (agent + fleet)
lists; write access is the `readwrite_*` lists only. **There is no wildcard** - only admins are
tenant-wide (via `None` lists); granting a non-admin "all agents" means listing every agent id
explicitly, so new agents are not auto-included. The scope is enforced uniformly: an out-of-scope agent is invisible on `GET /agents` and every
operation on it returns `404`; a **read-only** grant additionally rejects write commands (`POST
/jobs`) and approval creation with `403`, in _any_ policy mode. **Admins are always tenant-wide and
cannot be scoped.** An operator who creates an agent is automatically granted read-write access to
it (so a scoped operator can manage what they just created).

### Fleets

A **fleet** is a reusable **join token** for host agents, built for autoscaling groups of any flavour
(AWS ASGs, GCP MIGs, Azure VMSS, or any autoscaler): bake the token into the group's launch/instance
template (user-data or startup script) and every instance that scales in auto-enrolls as a
host agent, inheriting the fleet's **mode**, **tags**, and **grants**. Fleets are **host-only**
(k8s agents have one-agent-per-cluster identity). Re-enrollment is idempotent per
`(fleet_id, machine_fingerprint)`, so an instance rebooting reuses its record instead of
duplicating. Members that scale in are removed by the [reaper](ARCHITECTURE.md#fleets) after the
fleet's reap window (or by the agent calling `POST /agent/deregister` on shutdown).

Fleet agents are managed **through the fleet**: `PUT /tenant/agents/{id}/tags`, `/policy/mode`, and
`reissue-install-token` all return `409` for a member (mode/tags are inherited; enrollment is via the
join token). Rotating the member's **agent** token and acknowledging capabilities still work per-agent.
**Approvals are fleet-scoped too:** a member has no per-agent approvals - create them on the fleet (`POST
/tenant/approvals` with `fleet_id`) and every member inherits them. In `approved` mode, a member's sync
draws its allow-list from the fleet's approved commands, and a blocked write on a member raises a
**fleet-scoped** pending request (not a per-agent one).

### Approvals

| Method   | Path                             | Auth       | Rate limit | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |

| `GET`    | `/tenant/approvals`              | operator+  | 120/min    | List/search approvals in the tenant. Server-side filters: `?scope=agent\|fleet` (standalone-agent vs fleet approvals) `?agent_id=` `?fleet_id=` `?status=pending\|approved\|denied\|expired` `?type=host\|k8s` (by agent type; host approvals are structured `host_rule`s, k8s are `k8s_rule`s) `?q=` (case-insensitive `LIKE` over the command/rule text and requester). Response: `{"approvals": [...], "total": N, "limit": L, "offset": O}`; each row carries `scope` (`agent`\|`fleet`) plus `agent_hostname` or `fleet_name`. A **scoped operator** only sees approvals for the agents/fleets they can access; admins are tenant-wide.                                                                                                                                                                                                                                                                                                                                                        |
| `POST`   | `/tenant/approvals`              | developer+ | 60/min     | Create an approval for **one agent XOR one fleet** (exactly one of `agent_id`/`fleet_id`). **Developers** create a `pending` request; **operators/admins** create an `approved` record directly (and support bulk + `duration`). **Host agents/fleets** use a structured **host rule** `{bin, args[]}` (each arg a literal, the single-arg wildcard `*`, or a trailing `...` matching zero or more remaining args): single `{"agent_id"\|"fleet_id","host_rule":{"bin":"systemctl","args":["restart","*"]},"duration":"8h"}`, bulk `{...,"host_rules":[...]}`. A **command string** (`{...,"command"}` / `"commands":[...]`) is also accepted and **structured into a host rule** server-side; it must contain no shell operators, and is rejected (`400`) if it can't be structured (e.g. a glob arg that isn't a whole-arg `*`) - there is no raw command-string approval. **k8s agents** take a `kubectl` rule **or** a non-`kubectl` host rule: `k8s_rule`/`k8s_rules` `{"verb","resource","namespace","name"}` for kubectl, or `host_rule`/`host_rules` `{bin, args[]}` for a non-kubectl tool (`helm`, `flux`, …) - same positional model as host rules. k8s rule fields accept `*`; `verb` is required and must be a write verb - a single verb like `scale`/`delete`, a compound "double verb" like `rollout restart` or `auth reconcile`, or `*`. A host_rule whose `bin` is **not in the agent's reported execution allowlist** is rejected (`409`) - approving a binary the agent won't run is a no-op. Bulk is idempotent → `{"created":[...],"skipped":[...]}`. A **fleet member's `agent_id` is rejected (`409`)** - its approvals live on the fleet (every member inherits them); fleets are host-only so `k8s_rule` on a fleet is rejected. Writing to a fleet requires **read-write** access to it. |
| `PUT`    | `/tenant/approvals/{id}/approve` | operator+  | 120/min    | Approve (`pending`) or update duration (`approved`). Body: `{"duration": "permanent\|1h\|8h\|24h\|7d\|Nh\|Nd\|now"}`. Named presets are `1h`, `8h`, `24h`, `7d`; any other window is expressed as `Nh`/`Nd` (e.g. `30d`, `90d`). `duration=now` instantly expires an already-approved record.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `PUT`    | `/tenant/approvals/{id}/deny`    | operator+  | 120/min    | Deny a pending approval. Terminal - cannot be reversed.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `DELETE` | `/tenant/approvals/{id}`         | operator+  | 60/min     | Permanently delete an approval record. Removing an approved record takes effect on the next agent sync.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |

### Settings

Per-tenant retention windows, the fan-out blast-radius cap, and the staged-rollout wave policy. New tenants start on the platform defaults (no input required at creation); a tenant admin tunes them within bounds, and a platform admin can override beyond the bounds via `/admin/tenants/{id}/settings`.

| Method | Path               | Auth  | Rate limit | Description                                                                                                                                                                                                                                                                                                                                                                  |
| ------ | ------------------ | ----- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `GET`  | `/tenant/settings` | admin | 120/min    | This tenant's settings: `{settings, overrides, defaults, bounds, wave_policy}` where `settings` is the effective (in-force) value per key.                                                                                                                                                                                                                                   |
| `PUT`  | `/tenant/settings` | admin | 30/min     | Update settings **within tenant bounds**. Body: any of `approval_retention_days`, `job_retention_days`, `run_retention_days`, `audit_retention_days`, `agent_history_retention_days`, `fanout_cap` (positive ints within bounds; `null` clears a key back to the platform default), and `wave_policy` (see below). Retention keys are in days; `fanout_cap` is a host count. |

**Staged rollout (`wave_policy`).** Controls whether a fan-out runs in **waves** (of the fan-out cap) instead of all at once, per **read vs write** command. Shape: `{"tag": {"read"?, "write"?}, "fleet": {"read"?, "write"?}}` (the `fleet` branch is the tenant default for fleet runs; a fleet can override it via `PUT /tenant/fleets/{id}` with its own `{"read"?, "write"?}`). Each entry is `{"mode": "auto"|"manual", "on_failure": "stop"|"continue", "concurrency"?: int}` - `auto` releases the next wave automatically, `manual` pauses after every wave; `stop` pauses the rollout on a wave failure, `continue` keeps going. `concurrency` is the hosts-per-wave; it defaults to the fan-out cap and **can't exceed it** (`400` otherwise). A missing entry = not staged. Control a live staged run with `POST /tenant/runs/{run_id}/pause|resume|cancel`.

## Agent endpoints

Called by the agent process automatically - not intended for manual use.

| Method | Path                      | Auth          | Rate limit              | Description                                                                                                            |
| ------ | ------------------------- | ------------- | ----------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `POST` | `/agent/claim`            | install token | 120/min per IP          | One-time agent registration (also handles fleet join tokens)                                                           |
| `POST` | `/agent/sync`             | agent token   | 60/min per agent token  | Poll for jobs, record heartbeat, receive policy updates                                                                |
| `POST` | `/agent/jobs/{id}/result` | agent token   | 60/min per agent token  | Post command result. Body: `{machine_fingerprint, status, exit_code, stdout, stderr, duration_ms, blocked?, block_reason?, is_write?, stdout_truncated?, stderr_truncated?}`. The server re-caps `stdout`/`stderr` to 50 KB (forcing the `*_truncated` flag if it has to cut) and redacts secrets before storing. **`block_reason`** distinguishes an approvable block from an unfixable one: a blocked write normally raises a pending approval, but `block_reason: "sandbox_unavailable"` (fail-closed - the host has no kernel write protection) raises **none**, since approving can't satisfy a missing sandbox (acknowledge the host or move it to a Landlock kernel instead). |
| `POST` | `/agent/rotate-token`     | agent token   | 10/hour per agent token | Self-service token rotation (called automatically every 30 days)                                                       |
| `POST` | `/agent/deregister`       | agent token   | 60/min per agent token  | Host fleet member removes itself on machine shutdown (autoscaler scale-in). No-op (`409`) for non-fleet or k8s agents. |

These are **credential-only**: the agent never sends an `agent_id`. `claim` carries the **install token** (the backend resolves the agent by `install_token_hash`), a `machine_fingerprint`, and `type` (`host` or `k8s`), and returns the long-lived **agent token**. Every later call carries that agent token as the Bearer credential, and the backend resolves the agent by `agent_token_hash` - so identity is never taken from a client-supplied field. In `k8s` mode `sync` also reports the agent's effective RBAC (for acknowledge/drift in the console).

When the install token is a **fleet join token** (`fleet_…` prefix), `claim` enrolls the caller into that fleet instead: it mints or idempotently re-enrolls a host agent keyed on `(fleet_id, machine_fingerprint)`, inheriting the fleet's mode, tags, and grants. Fleet claims are rejected for `type: k8s`. `claim` is keyed by **IP** and rate-limited at 120/min so a NAT-shared autoscaling group can enroll many instances at once (the 256-bit tokens make the limit purely anti-DoS). `deregister` is the graceful counterpart - a host calls it only when the machine itself is shutting down (not on a service restart), so a scaled-in instance leaves its fleet at once rather than waiting for the reaper.

## Rate limits

Every endpoint is rate limited in the **Docker / FastAPI** deployment (via `slowapi`). The exact per-endpoint limit is listed in the `Rate limit` column of each table above. Lambda relies on API Gateway's built-in throttling and does not apply these per-route limits.

**Storage** - counters default to in-memory (per-process), which only enforces correctly on a single instance. To rate limit across multiple backend replicas, set `RATE_LIMIT_STORAGE_URI` to a shared store (e.g. `redis://host:6379`) on every replica - see [SELF_HOSTING.md → Running multiple replicas](SELF_HOSTING.md#running-multiple-replicas).

**Rate key** - limits are counted against the Bearer token when one is present: the tenant session token for `/tenant/*`; the API token for the CLI endpoints (`/me`, `/jobs`, `/agents`, `/approvals/*`); the platform-admin session token for `/admin/*`; the agent token for `/agent/sync`, `/agent/jobs/{id}/result`, `/agent/rotate-token`, and `/agent/deregister`. Where no usable token exists (`POST /admin/login`, `POST /tenant/login`, `POST /agent/claim`, `GET /health`), the key falls back to client IP.

**Limit tiers at a glance:**

| Tier                           | Typical limit          | Examples                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |

| Reads (all authenticated GETs) | 120/min                | Every read is 120/min - `GET /me`, `GET /jobs`, `GET /jobs/{id}`, `GET /agents`, `GET /agents/{id}`, `GET /agents/{id}/approved-commands`, `GET /approvals/pending`, `GET /tenant/approvals`, `GET /tenant/users`, `GET /tenant/users/{id}/agents`, `GET /tenant/fleets`, `GET /tenant/agent-versions`, `GET /tenant/agents/{id}/history`, `GET /tenant/api-tokens`, `GET /tenant/audit-logs`, `GET /admin/*`. ## Pagination

Paginated endpoints return `next_cursor` when more results exist:

```json
{
  "items": [...],
  "next_cursor": "MjAyNi0wNi0xN1QxMDowNTowMCswMDowMA=="
}
```

Pass `?cursor=<next_cursor>` on the next request. The cursor is absent on the last page. Default page size is 20, max 100.

**Audit-log endpoints differ:** they return the array under `logs` (not `items`), default to 100 per page (max 200), and `next_cursor` is the `created_at` timestamp of the last row:

```json
{
  "logs": [...],
  "next_cursor": "2026-06-17T10:05:00+00:00"
}
```

| `approval.denied`                    | Pending approval denied                                                                                     |
| `approval.expired`                   | Approved command instantly expired via `duration=now`                                                       |
| `approval.deleted`                   | Approval record deleted                                                                                     |
| `api_token.created`                  | Named API token created                                                                                     |
| `api_token.renamed`                  | Named API token renamed                                                                                     |
| `api_token.revoked`                  | Named API token revoked                                                                                     |
| `api_token.deleted`                  | Revoked API token record deleted                                                                            |
| `job.dispatched`                     | A single-agent job was created (`POST /jobs` - CLI `reach exec` or the console); metadata carries `agent_id`, `hostname`, `command`, `mode`, and `is_write`. Fan-outs use `run.dispatched` instead |
| `run.dispatched`                     | A fleet or tag fan-out dispatched (one event per **run**, not per member job); metadata carries `scope` (`fleet`/`tag`), the target, `command`, `dispatched` count, `wave_total`, and `is_write` |
| `run.paused` / `run.resumed`         | A staged (multi-wave) run paused / resumed                                                                  |
| `run.canceled`                       | A run canceled (remaining held waves dropped)                                                               |
| `tenant.settings_updated`            | Tenant admin updated their tenant's settings (within bounds); metadata lists changed `keys`                 |
| `tenant.settings_overridden`         | Platform admin overrode a tenant's settings (bounds bypassed); metadata lists changed `keys`               |

A fan-out records a single `run.dispatched` audit event when the run is created; its member jobs are correlated by the run's `resource_id` (their `run_id`), so there is no per-job audit spam. Dry-run previews and idempotent no-op retries do not write an audit event.

Platform admin logins are audited as `admin.login` (success) and `admin.login_failed` (wrong password), each capturing the source IP; the `ADMIN_PASSWORD`-not-configured (500) case is not logged. Approval reviews are written as `approval.approved`, `approval.denied`, or `approval.expired` (the latter when an approved record is instantly expired with `duration=now`); the approval record additionally carries `reviewed_by`, `reviewed_at`, and `status`. A natural time-based expiry (the scheduler sweeping a record past its `expires_at`) updates the record's status but does not write a separate audit action.
